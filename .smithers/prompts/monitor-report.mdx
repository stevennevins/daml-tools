# Render the monitor report (HTML)

You are the report step of the `monitor` workflow. Turn the captured snapshot,
the diagnosis, and any fix actions into ONE self-contained HTML document an
operator can open directly. HTML is the default deliverable here: prefer a clean,
scannable page over a wall of prose. Inline all CSS in a single `style` tag. No
JavaScript, no CDN links, no web fonts, no external images. One `.html` file.

## What the page must contain

Build one `section` per topic. Keep prose tight; lead with tables and lists.

1. **Header** — title, run id, a colored health badge (green healthy, amber
   blocked/stuck, red failed/overBudget), the reported status, and the one-line
   diagnosis summary.
2. **What it is waiting on / where it broke** — the blocker or root cause, with
   the single most important next action called out.
3. **Questions and answers** — a table of every question the run asked: prompt,
   answer (or a "pending" pill), who answered, node id. If none, say so.
4. **Approval gates** — a table of every gate: node id, decision (approved /
   denied / pending), note, who decided, when. If none, say so.
5. **Task outputs** — the notable node outputs, each as a labeled block with a
   tight summary (and the value when small). Escape angle brackets in content.
6. **Code diffs** — for tasks that changed files, the files touched and a
   representative excerpt in a `<pre>` block. If none, say so.
7. **Scores** — scorer results, if any.
8. **Recommended actions** — an ordered list; for each, the exact command in a
   `<code>` element and whether a human must run it.
9. **What the monitor fixed** — present only when fix actions exist: what was
   applied and whether the run resumed.
10. **Operator cheat-sheet** — a short reference so the reader can keep watching
    this run themselves. Include these commands verbatim (substitute the real run
    id), each with a one-line description:
    - `smithers ps` — list active / paused / recent runs.
    - `smithers inspect {props.runId} --watch` — live run state, nodes, gates.
    - `smithers logs {props.runId} -f` — tail the event log.
    - `smithers events {props.runId} --json` — full event history (NDJSON).
    - `smithers why {props.runId}` — why it is paused / blocked / failed.
    - `smithers output {props.runId} <nodeId>` — a task's output row.
    - `smithers diff {props.runId} <nodeId>` — a task's code diff.
    - `smithers approve {props.runId}` / `smithers deny {props.runId}` — clear a gate.
    - `smithers ui {props.runId}` — open the live custom UI for this run.
11. **Hot mode vs. normal mode** — a short explainer the reader can act on:
    - Normal mode (`smithers up <workflow>` / `smithers monitor`) runs the
      workflow as written; editing the source does not affect the in-flight run,
      and resuming after a source change is blocked because it is a different
      workflow.
    - Hot mode (`smithers up <workflow> --hot true`) applies edits to prompt
      wording and task bodies on the next render frame while finished tasks stay
      persisted. Use it while authoring or tuning a workflow.
    - Changing an output schema or a task id's shape needs a fresh run, not a hot
      edit. Keep task ids stable and data-derived so resume and hot reload
      preserve completed work.
    - This monitor run itself can be launched with `--hot` while you iterate on
      its prompts; the target run it watches is unaffected either way.

For any topic with no data, keep the section and state it plainly (for example,
"No approval gates in this run.") rather than inventing detail.

## HTML rules

- Emit ONE complete document starting with `<!DOCTYPE html>`.
- All styling in a single inline `style` block in the `head`. Dark, legible,
  monospace for ids/commands. Make the health badge color match the bucket.
- Escape literal angle brackets inside text/output/diff content so they render as
  text, not markup.
- No script tags, no external URLs. Self-contained only.

## Output

Return the chosen `title`, the full `html` document as one string, the run's
`health` bucket, and `sectionCount` (how many sections you produced).
