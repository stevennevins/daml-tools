Rough guide to what each is for:

- `implement` ‚Äî build a feature or change from a clear ask, with review.
- `research-plan-implement` ‚Äî research, then plan, then implement end-to-end.
- `review` ‚Äî review a specific diff or change before landing it.
- `plan` ‚Äî produce a plan or design without building yet.
- `research` ‚Äî gather and synthesize information about a topic.
- `grill-me` ‚Äî adversarially stress-test an idea, plan, or PRD with hard questions.
- `ralph` ‚Äî long-running autonomous loop that keeps working until a goal is met.
- `debug` ‚Äî investigate and fix the root cause of a defect in the user's own code.
- `audit` ‚Äî read-only sweep of a feature, surface, or area; deliverable is findings, not fixes.
- `create-workflow` ‚Äî author a brand-new Smithers workflow from a description.
- `context-doctor` ‚Äî diagnose and repair an agent's working context.
- `monitor-smithers` ‚Äî watchdog that detects stuck, blocked, failed, or over-budget runs.
- `triage-run` ‚Äî diagnose one failed or stuck Smithers run and recommend fix, rewind, retry, or escalate.
- `report-slideshow` ‚Äî turn a finished run into a self-contained HTML slideshow report.
- `eval-author` ‚Äî turn acceptance criteria into a runnable eval suite.
- `create-skill` ‚Äî author a new reusable agent skill.
- `extract-skill` ‚Äî harvest a reusable skill or workflow sketch from a finished run.
- `improve-test-coverage` ‚Äî raise test coverage until a target holds.

## Tie-breaking

Prefer the most SPECIFIC workflow that covers the ask over a general one:

- Diagnosing a failed or stuck **Smithers run** goes to `triage-run`, not
  `debug` ‚Äî `debug` is for defects in the user's own code.
- A read-only findings sweep is `audit`; reviewing a specific diff is `review`.
- Ongoing watching of many runs is `monitor-smithers`; one run's post-mortem is
  `triage-run`.
- If two workflows fit equally, pick the narrower one and list the broader one
  in `alternativeWorkflows`.

## Produce

1. `recommendedWorkflow` ‚Äî the single best-fit id from the list. Prefer the
   classifier's `recommendedWorkflow` when it is sound; override it only if a
   different listed workflow clearly fits better.
2. `why` ‚Äî why this workflow fits the task, naming the durable behaviour it
   provides (ordering, recovery, approval, or loops) that the task needs.
3. `alternativeWorkflows` ‚Äî other listed ids that could also fit, best-first; an
   empty array if nothing else is a reasonable second choice.
