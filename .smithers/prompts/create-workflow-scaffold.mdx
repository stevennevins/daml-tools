## How to write it

1. **Start from the closest example** named in the provisioning manifest — copy
   its structure, then adapt. Don't invent API; if unsure, re-read the example or
   `bunx smithers-orchestrator docs`.
2. **Workflow file header** — begin with the seed comments and the JSX pragma,
   exactly like the other workflows in `{props.workflowsDir}`:
   ```
   // smithers-source: seeded
   // smithers-metadata-version: 1
   // smithers-display-name: <Title Case Name>
   // smithers-description: <one sentence: what the workflow does>
   // smithers-tags: <comma-separated tags, e.g. ops, reporting>
   /** @jsxImportSource smithers-orchestrator */
   ```
3. **Imports** — `import { createSmithers } from "smithers-orchestrator"`,
   `import { z } from "zod/v4"`, `import { agents } from "../agents"`, any
   components from `../components`, and each prompt from `../prompts/<name>.mdx`.
4. **Schemas** — define the Zod input schema and one schema per task output.
   Call `createSmithers({ input: inputSchema, <table>: <schema>, ... })` and
   destructure only the components you use (`Workflow`, `Task`, `Sequence`,
   `Branch`, `Loop`, `Approval`, `smithers`, `outputs`).
5. **Graph** — `export default smithers((ctx) => ( <Workflow name="<id>"> … ))`,
   matching the design's `graphShape`. Gate later stages on earlier outputs with
   `ctx.outputMaybe("table", { nodeId })`, exactly like the existing workflows.
6. **Prompts** — author each `.mdx` from the design's `prompts` list. Pass props
   from the task and end every prompt with:
   ```
   REQUIRED OUTPUT:
   {props.schema}
   ```
   `.mdx` prompt files are parsed as MDX modules. Wrap JSX/HTML-like prose examples
   such as `<Parallel maxConcurrency={N}>` in inline backticks or fenced code blocks;
   otherwise MDX treats the example as JSX and can break the default export.
7. **Agents** — only add a new provider/pool to `../agents.ts` if the design
   truly needs one that doesn't exist; otherwise reuse the existing pools.
8. **Skills** — if the design relies on a worker skill the provisioning step
   marked `recommended` (not yet installed), create a concise skill file under
   the local skills dir so the workflow's agents have it.

Keep it idiomatic and minimal — match the style of the neighbouring files. Do not
add dead code, placeholders, or "TODO" stubs. The next step renders the graph to
verify it compiles, so make it actually load.
