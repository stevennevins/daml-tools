# Advise on the context contract diagnosis

The deterministic checker has already diagnosed a context contract. Your job is
to turn its findings into concrete, actionable advice. Do NOT re-run the checks,
re-grade severities, or invent findings ‚Äî every fix must trace back to exactly
one issue the checker reported.

## Diagnosis summary

{props.summary}

Health score: {props.score} / 100

## Issues

{`\`\`\`json
${JSON.stringify(props.issues, null, 2)}
\`\`\``}

Each issue has a `check` id, a `severity` (`ok`, `warning`, `error`, or `info`),
and a `message`.

## What to produce

1. For every issue whose severity is NOT `ok`, emit exactly one object
   `{check, fix}`:
   - `check` ‚Äî the issue's check id, copied VERBATIM. Never rename, abbreviate,
     or merge ids.
   - `fix` ‚Äî one imperative sentence naming the exact contract field to add or
     change.
2. One non-ok issue ‚Üí one object. Do not emit fixes for checks that passed, and
   do not split one issue into several fixes.
3. Order `fixes` by severity: `error` first, then `warning`, then `info`.
   Within the same severity, keep the checker's issue order.
4. If every issue is `ok`, return an empty `fixes` array and say the contract
   is healthy in your `summary`.

## What a good fix names, per check

- `hasGoal` ‚Äî add a `goal` stating the intended outcome.
- `hasOutputSpec` ‚Äî add an `outputSpec` naming the produced artifact.
- `hasAcceptanceCriteria` ‚Äî add `acceptanceCriteria` as `{text, blocking,
  verification}` objects.
- `allBlockingCriteriaHaveVerification` ‚Äî give each blocking criterion a
  `verification` (convert plain-string criteria to objects so they can declare
  one).
- `allRequiredInputsHaveSource` ‚Äî add a `source` to each required input.
- `allSideEffectsHaveApproval` ‚Äî add an `approval` to each side effect (convert
  plain-string side effects to objects so they can declare one).
- `reportSpecExists` ‚Äî add a `reportSpec` describing how results are reported.
- `parse` ‚Äî make the contract valid JSON with an object at the top level.

Note: the checker also flags acceptance criteria and side effects written as
plain strings ‚Äî a string cannot declare `blocking`, `verification`, or
`approval`, so the fix is to convert it to an object that can.

## Worked examples

```
Issue: {"check": "hasGoal", "severity": "error", "message": "Missing a non-empty `goal` describing the intended outcome."}
Fix:   {"check": "hasGoal", "fix": "Add a goal field stating the outcome this contract is meant to achieve."}

Issue: {"check": "allSideEffectsHaveApproval", "severity": "warning", "message": "2 side effect(s) have no `approval` gate."}
Fix:   {"check": "allSideEffectsHaveApproval", "fix": "Add an approval gate to each of the 2 unguarded side effects, converting plain-string entries to objects."}

Every issue ok ‚Üí {"fixes": [], "summary": "Contract is healthy; no fixes needed."}
```

Keep each fix to one sentence. Ground it in the issue's `message` ‚Äî reuse its
counts and field names; do not invent details the checker did not report.

## Output

Return `fixes` (one `{check, fix}` object per non-ok issue, ordered
error ‚Üí warning ‚Üí info) and a one-line `summary` of the overall remediation.
