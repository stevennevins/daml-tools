# Release the workspace

Use this flow to publish Rust crates, the `@daml-tools/lint-plugin` npm
package, cargo-npm CLI packages, and GitHub release artifacts.

`daml-parser` releases publish the Rust crate and release notes only. They do
not publish npm packages or GitHub binary assets.

## Check release authentication

Confirm the required GitHub secrets exist:

```sh
gh secret list --repo stevennevins/daml-tools
```

Required secrets:

| Secret | Purpose |
|--------|---------|
| `CARGO_REGISTRY_TOKEN` | Lets release-plz publish crates to crates.io. |
| `RELEASE_PLZ_TOKEN` | Lets release-plz-created tags trigger follow-up workflows. |

Optional bootstrap secret:

| Secret | Purpose |
|--------|---------|
| `NPM_TOKEN` | Emergency/bootstrap only: lets a manually dispatched npm publish workflow use a short-lived npm granular access token when trusted publishing cannot create or recover a package. Remove it after use. |

Normal npm publishing uses npm trusted publishing, so tag-triggered workflows do
not need an npm token. Configure trusted publishing on npmjs.com for every
published npm package:

- `@daml-tools/lint-plugin`
- `@daml-tools/daml-lint`
- `@daml-tools/daml-lint-darwin-arm64`
- `@daml-tools/daml-lint-linux-arm64`
- `@daml-tools/daml-lint-linux-x64`
- `@daml-tools/daml-lint-win32-x64`
- `@daml-tools/daml-fmt`
- `@daml-tools/daml-fmt-darwin-arm64`
- `@daml-tools/daml-fmt-linux-arm64`
- `@daml-tools/daml-fmt-linux-x64`
- `@daml-tools/daml-fmt-win32-x64`

The CLI npm packages are generated by cargo-npm from `[package.metadata.npm]`
in `crates/daml-lint/Cargo.toml` and `crates/daml-fmt/Cargo.toml`. There are no
committed `crates/*/npm` package directories to update.

Supported CLI npm targets:

| npm package suffix | Rust target |
|--------------------|-------------|
| `linux-x64` | `x86_64-unknown-linux-gnu` |
| `linux-arm64` | `aarch64-unknown-linux-gnu` |
| `darwin-arm64` | `aarch64-apple-darwin` |
| `win32-x64` | `x86_64-pc-windows-msvc` |

For a brand-new npm package, npm may require package creation before trusted
publishing can be configured. Add a temporary `NPM_TOKEN` granular access token
secret, dispatch `npm-publish.yml` manually from the release tag with
`use_npm_token=true`, configure trusted publishing for the new package names,
then remove the token. Normal tag workflows ignore `NPM_TOKEN`.

Do not add `--provenance` while this repository is private. npm trusted
publishing works from this repo, but npm rejects provenance bundles whose
GitHub source repository is private.

## Review the release PR

Release-plz opens a `chore: release` PR after semver-relevant changes land on
`main`. Normal `docs:`, `ci:`, and `chore:` commits do not prepare releases;
`feat:`, `fix:`, `perf:`, `refactor:`, `security:`, and breaking `!` commits
do.

For `daml-lint` release PRs, the Release-plz workflow syncs only the committed
lint-plugin npm metadata:

- `crates/daml-lint/package.json`
- `crates/daml-lint/package-lock.json`
- `crates/daml-lint/lint-plugin/package.json`
- `crates/daml-lint/lint-plugin/templates/project/package.json`

Release-plz does not sync CLI npm metadata. The `@daml-tools/daml-lint`,
`@daml-tools/daml-fmt`, and per-platform CLI packages are generated at publish
time from the Cargo package version and cargo-npm metadata.

Before merging the release PR, verify:

- Rust crate versions and changelogs are correct.
- The lint-plugin package files above match the `daml-lint` crate version when
  `daml-lint` is being released.
- The starter template depends on the new `@daml-tools/lint-plugin` version.
- CI is green, including the `npm CLI packaging` job.

CI runs `cargo npm generate --infer-targets` for both CLI crates, validates the
generated wrapper `bin`, `engines`, and optional dependencies, installs the
generated host package, and runs the generated binary.

## Merge the release PR

Merge the release PR into `main`.

This triggers the release flow:

1. `Release-plz` publishes changed Rust crates to crates.io.
2. Release-plz creates GitHub tags and releases.
3. `daml-lint-v*` tags publish the `@daml-tools/daml-lint-*` platform packages,
   then the `@daml-tools/daml-lint` wrapper package.
4. `daml-lint-v*` tags publish `@daml-tools/lint-plugin` to npm.
5. `daml-fmt-v*` tags publish the `@daml-tools/daml-fmt-*` platform packages,
   then the `@daml-tools/daml-fmt` wrapper package.
6. `daml-lint-v*` and `daml-fmt-v*` tags upload CLI archives and checksums to
   GitHub releases.

Plain `X.Y.Z` npm package versions publish under `latest`. Semver prerelease
versions publish under `next`.

Release-plz creates a follow-up release PR only after the release job finishes.

## Prerelease CLI npm packages

Use a prerelease to test CLI npm distribution under `next` without moving
`latest`. This does not publish crates to crates.io and does not run
release-plz.

1. Choose a never-published version, such as `0.2.6-rc.0`.
2. On a branch, bump exactly the crate being tested and refresh `Cargo.lock`:

   ```sh
   cargo set-version -p daml-fmt 0.2.6-rc.0
   cargo build
   cargo build --locked
   ```

   If `cargo set-version` is not installed, edit the crate `Cargo.toml` version
   and use the same `cargo build` commands to update and verify `Cargo.lock`.

3. For a `daml-lint` prerelease that also tests `@daml-tools/lint-plugin`, sync
   the lint-plugin metadata:

   ```sh
   (cd crates/daml-lint && node tools/sync-lint-plugin-version.mjs)
   ```

4. Validate locally, commit, push the branch, create the tag at that commit,
   and push the tag:

   ```sh
   git tag daml-fmt-v0.2.6-rc.0
   git push origin HEAD
   git push origin daml-fmt-v0.2.6-rc.0
   ```

5. If trusted publishing cannot create a first-time package, manually dispatch
   the npm workflow from the release tag with `use_npm_token=true`:

   ```sh
   gh workflow run npm-publish.yml --repo stevennevins/daml-tools \
     --ref daml-fmt-v0.2.6-rc.0 \
     -f tag=daml-fmt-v0.2.6-rc.0 \
     -f use_npm_token=true
   ```

6. Watch `npm-publish.yml` and `release-artifacts.yml`. A semver prerelease
   publishes CLI npm packages under `next`, keeps `latest` unchanged, and marks
   the GitHub release as a prerelease.

Verify the prerelease:

```sh
npm view @daml-tools/daml-fmt dist-tags --prefer-online
npm view @daml-tools/daml-fmt-linux-x64@0.2.6-rc.0 version --prefer-online
tmp="$(mktemp -d)" && cd "$tmp"
npm init -y >/dev/null
npm install --save-dev @daml-tools/daml-fmt@next --prefer-online
npx --no-install daml-fmt --version
```

If a prerelease publish partially fails after any npm package version is
created, rerun the workflow with `use_npm_token=true` only after checking the
registry state. The workflow skips generated package versions that already
exist, publishes the missing platform packages, then publishes the wrapper.

## Verify the release

Check workflow status:

```sh
gh run list --repo stevennevins/daml-tools --limit 12
```

If `gh workflow list --all` still shows `Npm Registry Smoke` as active, treat it
as stale GitHub Actions metadata. `origin/main` no longer contains
`.github/workflows/npm-registry-smoke.yml`, so it is not a current release gate.

Check crates.io:

```sh
cargo search daml-parser --limit 3
cargo search daml-lint --limit 3
cargo search daml-fmt --limit 3
```

Check npm:

```sh
npm view @daml-tools/lint-plugin version dist-tags time.modified --prefer-online
npm view @daml-tools/daml-lint version dist-tags time.modified --prefer-online
npm view @daml-tools/daml-fmt version dist-tags time.modified --prefer-online
```

Check one installed wrapper on the current platform:

```sh
tmp="$(mktemp -d)" && cd "$tmp"
npm init -y >/dev/null
npm install --save-dev @daml-tools/daml-lint@latest --prefer-online
npx --no-install daml-lint --version
```

Check GitHub release assets:

```sh
gh release view daml-lint-vX.Y.Z --repo stevennevins/daml-tools --json assets,url
gh release view daml-fmt-vX.Y.Z --repo stevennevins/daml-tools --json assets,url
```

Each CLI GitHub release should have `daml-tools-*` archives and matching
`.sha256` files:

- `daml-tools-x86_64-unknown-linux-gnu.tar.gz`
- `daml-tools-aarch64-unknown-linux-gnu.tar.gz`
- `daml-tools-aarch64-apple-darwin.tar.gz`
- `daml-tools-x86_64-pc-windows-msvc.zip`

## Recover a failed npm publish

If the `Publish npm Packages` workflow fails after the tag exists and no npm
package versions were created, rerun it with the release tag:

```sh
gh workflow run npm-publish.yml --repo stevennevins/daml-tools --ref daml-lint-vX.Y.Z -f tag=daml-lint-vX.Y.Z
gh workflow run npm-publish.yml --repo stevennevins/daml-tools --ref daml-fmt-vX.Y.Z -f tag=daml-fmt-vX.Y.Z
```

The workflow checks out the requested tag, verifies the Cargo package version
matches the tag, builds the tagged CLI binary for each supported platform,
generates cargo-npm packages, publishes missing platform packages first,
publishes the wrapper if missing, and smoke-installs the wrapper from npm.

If the wrapper package version already exists on npm, the CLI publish job skips
publishing and still runs the smoke install. If some platform packages exist but
the wrapper does not, inspect the registry state before rerunning so the missing
package set is clear in the recovery notes.

Only set `use_npm_token=true` for emergency/bootstrap recovery when trusted
publishing cannot create or recover a package. Leave it unset for normal trusted
publishing reruns.

## Recover failed release artifacts

If release artifact upload fails after a tag exists, rerun the artifact workflow
with that tag:

```sh
gh workflow run release-artifacts.yml --repo stevennevins/daml-tools --ref daml-lint-vX.Y.Z -f tag=daml-lint-vX.Y.Z
gh workflow run release-artifacts.yml --repo stevennevins/daml-tools --ref daml-fmt-vX.Y.Z -f tag=daml-fmt-vX.Y.Z
```

The artifact workflow checks out the requested tag and uploads archives to that
release.
